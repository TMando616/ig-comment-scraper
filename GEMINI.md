# AI Coding Rules & Conventions

## 1. 開発プロセス
- **README.md の優先確認**: 毎回、作業を開始する前に必ず `README.md` を確認し、現在の詳細な実装状況を把握すること。
- **README.md の継続的更新**: 機能修正や追加を行うたびに、その詳細な実装内容（ロジック、依存関係、変更点など）を `README.md` の「詳細な実装内容」セクションに追記すること。これにより、常に最新の状態をドキュメントから把握できるようにする。
- **テストの優先実装**: 新機能の追加やバグ修正を行う際は、必ず対応するテストコード（正常系・異常系）を `tests/` ディレクトリ配下に追加または更新すること。

## 2. コミットメッセージ
- 日本語を使用すること。
- プレフィックス（feat:, fix:, refactor:, docs:, chore:, ci: など）を付けること。
- 細かい粒度でコミットすること。

## 3. コーディング規約
- **コメント**: 処理の意図がわかるように適宜日本語でコメントを残すこと。
- **モジュール化**: 責務ごとにファイルを分割し、再利用性と可読性を高めること（例：`main.py`, `scraper.py`, `spreadsheet.py`）。
- **型ヒント**: Pythonコードには必ず型ヒント（Type Hinting）を記述すること。
- **機密情報**: ログインID、パスワード、APIキーなどの機密情報は、必ず `.env` ファイルから読み込む設計にし、コード内にハードコードしないこと。
- **エラーハンドリング**: スクレイピング処理には適切な `try-except` を実装し、エラー発生時は原因がわかるようにログ（または print）を出力すること。

## 4. テストと CI/CD
- **テストフレームワーク**: `pytest` および `pytest-playwright` を採用する。
- **ディレクトリ構成**: 全てのテストコードは `tests/` ディレクトリ内に配置し、共通のフィクスチャ（セッション共有など）は `tests/conftest.py` で管理すること。
- **CI/CDの運用**: GitHub Actions を利用し、`main` ブランチへのプッシュまたはプルリクエスト時に自動テストが Docker 環境で実行される仕組みを維持すること。
- **品質管理**: CI でのテストパスをマージの前提条件とし、回帰バグの防止に努めること。
